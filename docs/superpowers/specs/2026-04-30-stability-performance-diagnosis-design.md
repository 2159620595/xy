# 全链路稳定性与性能诊断首轮设计

## 背景

当前项目是一个前后端一体仓库，包含以下主要部分：

- `backend/`：主后端、聚店相关后端、网盘相关后端与静态托管
- `frontend/`：当前整合版前端源码
- 最近历史中已出现性能优化与回退恢复，说明项目既存在性能诉求，也需要控制改动风险

用户目标不是只拿一份分析报告，而是希望对当前项目进行实际优化。但“性能、稳定性、前端体验、代码结构全部一起做”范围过大，直接实施会造成回归风险和目标失焦。

因此，本轮仅聚焦一个可控子目标：先做全链路稳定性与性能诊断，再落地 1 到 3 个低风险、高收益、可验证的优化项。

## 目标

本轮设计的目标如下：

1. 识别当前项目中最值得优先处理的稳定性与性能问题
2. 诊断范围覆盖前端请求链路、主后端、聚店流程、网盘流程
3. 基于诊断结果，只落地 1 到 3 个低风险问题修复或优化
4. 为后续更大范围的体验优化或结构优化提供有依据的下一步方向

## 非目标

本轮明确不做以下事项：

- 大规模重构
- 跨多个子系统的架构迁移
- 无证据支撑的“猜测式性能优化”
- 为未来可能需求预先增加抽象层
- 全面覆盖所有发现的问题

## 推荐方案

候选方案有三种：

1. 纯诊断，不改代码
2. 诊断加小步修复
3. 以性能体感为主的集中提速

本轮采用方案 2，即“诊断加小步修复”。

采用原因：

- 比纯诊断更能产生实际收益
- 比集中提速风险更低，更适合当前仓库状态
- 能在不扩大改动面的前提下，验证诊断结论是否正确

## 诊断范围

诊断覆盖以下链路与模块：

- `frontend/`：页面加载、请求发起、路由切换、构建产物相关问题
- 主后端：`backend/Start.py`、`backend/reply_server.py`、主业务相关接口与任务处理路径
- 聚店流程：`backend/judian/` 与前端聚店页面、聚店接口调用链路
- 网盘流程：`backend/netdisk/` 与前端网盘页面、网盘接口调用链路

## 诊断方法

第一轮以静态排查为主，不默认进行全面压测。诊断时重点关注以下信号：

- 慢接口：超时配置过保守、串行流程过长、重复请求、缺少局部缓存
- 不稳定点：异常处理缺失、失败后行为不清晰、登录态依赖脆弱、边界情况容易失败
- 前端体感问题：首屏负担过重、路由切换成本高、包体或资源组织异常
- 高风险模块：近期频繁改动、职责过多、文件过大、耦合过强

如果静态排查不足以支撑结论，再补充最小必要的运行验证，而不是先做全面压测。

## 首轮修复边界

首轮只允许落地 1 到 3 个问题修复，且必须同时满足以下条件：

- 改动小
- 回归面可控
- 收益明确
- 可以独立验证

优先选择的优化类型包括：

- 明显不合理的超时或重试配置
- 能直接减少重复请求或串行等待的小范围改动
- 明确的异常分支补强
- 不改变业务规则的前端加载优化

## 产出物

本轮产出分为两部分：

### 1. 诊断清单

至少包含以下内容：

- 最慢的候选链路
- 最不稳的候选链路
- 问题大致所在模块或文件
- 每个问题的影响范围
- 每个问题的修复优先级和理由

### 2. 首批优化落地

针对选中的 1 到 3 个问题，明确记录：

- 为什么优先处理它
- 做了什么改动
- 预期改善点是什么
- 如何验证没有引入明显副作用

## 验证标准

本轮任务完成的判断标准如下：

1. 明确识别出一组高优先稳定性或性能问题，而不是泛泛而谈
2. 至少落地 1 到 3 个低风险优化点
3. 每个优化点都有可复查的验证依据
4. 不引入新的明显错误
5. 不借机做无关重构

## 执行顺序

1. 盘点入口、关键模块与主要链路
2. 锁定最可能的稳定性与性能问题
3. 选出 1 到 3 个低风险高收益点实施
4. 做必要验证并输出结果总结

## 风险控制

为降低本轮优化风险，执行时遵循以下原则：

- 每次只做与诊断结论直接相关的改动
- 不因“顺手”修改相邻模块
- 若发现问题需要大改，记录为后续候选项，不在本轮展开
- 优先保持现有代码风格与模块边界

## 后续衔接

如果本轮完成后仍存在明显的前端体验或结构问题，可以基于本轮结论继续拆出后续子任务，例如：

- 前端首屏与路由加载专项
- 聚店兑换稳定性专项
- 网盘链路超时与失败恢复专项
- 高耦合模块的定向收敛

本设计仅覆盖第一轮“全链路诊断优先”的稳定性与性能优化工作。
