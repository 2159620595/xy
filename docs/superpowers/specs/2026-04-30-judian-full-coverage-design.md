# 聚点全业务诊断与测试补齐首轮设计

## 背景

当前仓库中的聚点业务同时分布在前后端多个位置：

- `frontend/src/api/judian.ts`：聚点管理端与公开接口调用入口
- `frontend/src/utils/judian-auth.ts`：聚点会话校验、同步与登录态恢复
- `frontend/src/views/judian/`：聚点管理页面与公开兑换页面
- `backend/judian/judian_backend/__init__.py`：聚点后端接口、远端调用、批量任务与公共流程
- `backend/judian/scripts/test-vip.js`：聚点远端调用与业务动作脚本

当前目标不是直接对聚点系统做大规模重构，而是先建立“主要业务可验证、问题可记录、修复可追踪”的首轮基线。

用户要求本轮覆盖前后端聚点主要业务，并同时包含自动化测试与冒烟测试。

## 目标

本轮设计的目标如下：

1. 盘点聚点前后端主要业务链路，并形成清晰的覆盖清单
2. 为聚点主要业务补齐首轮自动化测试入口
3. 为聚点主要接口和关键动作整理并执行冒烟测试
4. 在补测试过程中，修复阻塞测试建立或明显影响主链路的高风险问题
5. 输出与真实验证结果一致的聚点专项诊断与测试报告

## 非目标

本轮明确不做以下事项：

- 对聚点前后端做整体重构
- 为所有远端依赖建立完整离线仿真环境
- 为每个页面和每个接口追求高覆盖率测试
- 顺带修改非聚点模块
- 在没有测试或验证证据的情况下做猜测式优化

## 推荐方案

候选方案有三种：

1. 诊断优先，测试分层补齐，修复只做阻塞项
2. 前后端并行补测试，边测边修
3. 先做冒烟测试，再回补自动化测试

本轮采用方案 2，即“前后端并行补测试，边测边修”。

采用原因：

- 更接近“覆盖聚点全部主要业务”的目标
- 能尽早暴露前后端接口契约和页面状态问题
- 比先做大修复更稳，因为修复动作受测试建立过程约束

## 业务范围

首轮覆盖以下五类聚点链路：

### 1. 前端鉴权与会话链路

- `getJudianToken()`
- `verifyStoredJudianSession()`
- `ensureJudianSession()`
- 401 后的会话恢复和跳转行为

### 2. 前端管理端 API 链路

- 账号管理
- 卡密管理
- 卡片管理
- 导入相关接口
- 批量相关接口

### 3. 前端公开兑换链路

- 兑换码查询
- 扫码解锁
- 批量预览
- 批量购买
- 取消任务
- 页面状态展示

### 4. 后端聚点接口链路

- FastAPI 路由入口
- 请求模型
- 返回结构
- 鉴权边界
- 缓存与批量任务相关接口

### 5. 后端远端调用链路

- 登录
- 用户信息
- 套餐列表
- 下单
- 查单
- 支付
- 批量处理
- 状态轮询

## 测试策略

本轮采用“自动化测试 + 冒烟测试”并行的方式。

### 自动化测试

自动化测试分为前端和后端两层。

前端优先覆盖：

- `frontend/src/utils/judian-auth.ts`
- `frontend/src/api/judian.ts`
- `frontend/src/views/judian/RedeemView.vue` 的关键状态切面

后端优先覆盖：

- `backend/judian/judian_backend/__init__.py` 中适合本地验证的请求模型、关键接口和关键函数
- 与远端依赖耦合较深的部分，以最小隔离方式建立可重复执行的测试入口

自动化测试目标不是首轮穷尽所有分支，而是保证每条主链路至少有一类可重复执行的本地验证。

### 冒烟测试

冒烟测试依托 `backend/judian/scripts/test-vip.js`。

首轮至少整理并评估以下动作：

- `smoke`
- `login`
- `list`
- `buy`
- `status`
- `autopay`
- `batch`

冒烟测试用于补足真实远端依赖和真实业务编排覆盖，输出应能够直接写入报告。

## 测试矩阵

### 前端自动化测试

`judian-auth.ts` 首轮至少验证：

- 空 token 时的行为
- 已有 token 的校验行为
- 校验失败后的会话清理
- 会话引导失败时的返回
- 401 时主登录态清理与跳转行为

`judian.ts` 首轮至少验证：

- 鉴权客户端和公共客户端的超时配置
- 登录相关接口使用更长超时
- 公共接口走公共客户端
- 关键接口配置不回退到裸请求

聚点页面首轮至少验证：

- 加载态
- 错误态
- 成功态中的关键显示
- 关键按钮禁用条件
- 批量任务入口或状态展示的最小切面

### 后端自动化测试

后端首轮至少验证：

- 关键请求模型的基础约束
- 主要接口的最小响应行为
- 关键失败分支的状态码或错误结构
- 可隔离的远端调用封装逻辑

对强依赖真实远端账号、真实订单或真实支付的行为，不强行做纯离线高保真测试，而是记录为冒烟验证项。

### 冒烟测试

首轮冒烟测试结果需要明确记录：

- 命令
- 前置条件
- 成功标准
- 实际结果
- 失败点
- 是否可重复执行

## 执行顺序

1. 盘点聚点页面、前端 API、后端接口和脚本入口
2. 建立链路清单与测试矩阵
3. 先补最小失败测试，确认失败原因正确
4. 只做让测试转绿所需的最小修复
5. 整理并执行首轮冒烟命令
6. 回写聚点专项诊断与测试报告

## 改动边界

本轮允许的代码改动仅限以下两类：

- 阻塞测试建立的问题修复
- 明显影响主业务链路稳定性的高风险问题修复

本轮不做：

- 大文件顺手拆分
- 与聚点无关的前后端优化
- 抽象出新的通用框架
- 成批修改页面样式或交互设计

## 产出物

本轮交付产物如下：

1. 聚点专项设计文档
2. 聚点专项诊断与测试报告
3. 前端聚点自动化测试增量
4. 后端聚点自动化测试增量
5. 聚点冒烟命令清单与执行记录

## 验证标准

本轮任务完成的判断标准如下：

1. 聚点主要业务链路都有清单
2. 聚点前端至少新增一组自动化测试
3. 聚点后端至少新增一组自动化测试
4. 至少整理并执行一组聚点冒烟验证
5. 报告内容能与真实命令输出、测试结果和修复项对上
6. 不借机做无关重构

## 风险与约束

首轮存在以下明确约束：

- `frontend/src/views/judian/RedeemView.vue` 体量较大，只适合切关键状态切面建立测试，不适合首轮追求高覆盖
- 后端聚点远端调用依赖外部服务，部分能力只能以受限冒烟方式验证
- 如果现有后端测试基础薄弱，首轮只补最小测试入口，不顺带迁移整套测试框架
- 真实账号、真实订单、真实支付相关能力可能需要手工前置条件，本轮需如实记录限制条件

## 后续衔接

如果首轮完成后仍发现聚点存在较多未解问题，后续任务按链路继续拆分：

- 聚点公开兑换稳定性专项
- 聚点远端下单与支付专项
- 聚点管理端页面与接口契约专项
- 聚点会话与鉴权恢复专项

本设计仅覆盖首轮“聚点全业务诊断与测试补齐”的实施边界。
